  <what-built>
    The Bicep update in Task 2 creates a Microsoft.Authorization/roleAssignments resource. The current GitHub Actions OIDC service principal has Contributor on the resource group but NOT Microsoft.Authorization/roleAssignments/write. The Bicep redeploy will fail with "AuthorizationFailed" unless the SP is granted permission to write role assignments before the deployment runs.
  </what-built>
  <how-to-verify>
    Grant the permission using one of these two approaches (pick one):

    **Option A — Azure Portal:**
    1. Navigate to the resource group `ev-catftx1web-rg`
    2. Click "Access control (IAM)" in the left panel
    3. Click "Add role assignment"
    4. Role tab: search for "Role Based Access Control Administrator" and select it
    5. Members tab: select "User, group, or service principal", find the SP named `catftx1web-github-sp` (or whatever APP_NAME prefix was used)
    6. Review + assign

    **Option B — Azure CLI (from a terminal authenticated as Owner):**
    ```bash
    # Get the APP_ID of the GitHub Actions SP
    APP_ID=$(az ad app list --display-name "catftx1web-github-sp" --query "[0].appId" -o tsv)
    SUBSCRIPTION=$(az account show --query id -o tsv)
    RG="ev-catftx1web-rg"

    az role assignment create \
      --assignee "$APP_ID" \
      --role "Role Based Access Control Administrator" \
      --scope "/subscriptions/${SUBSCRIPTION}/resourceGroups/${RG}"
    ```

    After running either option, verify the assignment appears:
    ```bash
    az role assignment list --assignee "$APP_ID" --scope "/subscriptions/${SUBSCRIPTION}/resourceGroups/${RG}" --query "[].roleDefinitionName" -o tsv
    ```
    Expected output includes both "Contributor" and "Role Based Access Control Administrator".

  <action>
    Rewrite infra/main.bicep to replace admin credential usage with system-assigned managed identity. The changes are:

    **1. Remove the acrCreds variable (line 39):**
    Delete: `var acrCreds = acr.listCredentials()`

    **2. Change ACR adminUserEnabled from true to false:**
    ```bicep
    properties: {
      adminUserEnabled: false
    }
    ```

    **3. Add identity block to the webapp resource (after `location: location` line):**
    ```bicep
    identity: {
      type: 'SystemAssigned'
    }
    ```

    **4. Remove the three DOCKER_REGISTRY_SERVER_* app settings from webapp siteConfig.appSettings:**
    Remove these three objects from the appSettings array:
    - `{ name: 'DOCKER_REGISTRY_SERVER_URL', value: 'https://${acr.properties.loginServer}' }`
    - `{ name: 'DOCKER_REGISTRY_SERVER_USERNAME', value: acrCreds.username }`
    - `{ name: 'DOCKER_REGISTRY_SERVER_PASSWORD', value: acrCreds.passwords[0].value }`
    Keep: `WEBSITES_ENABLE_APP_SERVICE_STORAGE` and `WEBSITES_PORT`.

    **5. Add a variable for the AcrPull role GUID above the ACR resource:**
    ```bicep
    var acrPullRoleId = '7f951dda-4ed3-4680-a7ca-43fe172d538d'
    ```

    **6. Add the AcrPull role assignment resource after the webapp resource:**
    ```bicep
    resource acrPullAssignment 'Microsoft.Authorization/roleAssignments@2022-04-01' = {
      name: guid(acr.id, webapp.id, acrPullRoleId)
      scope: acr
      properties: {
        roleDefinitionId: subscriptionResourceId(
          'Microsoft.Authorization/roleDefinitions',
          acrPullRoleId
        )
        principalId: webapp.identity.principalId
        principalType: 'ServicePrincipal'
      }
    }
    ```

    **7. Add the webappConfig resource to enable managed identity image pulls (after acrPullAssignment):**
    ```bicep
    resource webappConfig 'Microsoft.Web/sites/config@2023-12-01' = {
      parent: webapp
      name: 'web'
      properties: {
        acrUseManagedIdentityCreds: true
      }
    }
    ```

    The `guid(acr.id, webapp.id, acrPullRoleId)` function generates a deterministic GUID — idempotent across Bicep redeployments (will not create a duplicate role assignment on redeploy).

    Do not change the outputs section, the plan resource, or the param declarations. Do not change linuxFxVersion — it still uses `:latest` for initial deploy; the CI workflow continues to deploy via `:sha` tag using `azure/webapps-deploy@v3`.

    Deployment sequencing note (per D-10): Bicep deploys resources in dependency order. Because `acrPullAssignment` references `webapp.identity.principalId`, Bicep creates the managed identity before granting the role. The `webappConfig` (which removes old credentials via `acrUseManagedIdentityCreds: true`) depends on `parent: webapp`, so App Service restarts after the role is already assigned. This minimises the credential-gap window.

    Post-deploy note: After `az deployment group create` completes, allow 2-5 minutes for RBAC propagation before triggering a container restart. The App Service log may briefly show `unauthorized: authentication required` — this is transient.
