  <action>
    In nginx.conf, add one line immediately after the existing three security headers (after the `Referrer-Policy` line, before the blank line that precedes `# Gzip`).

    The three existing headers are:
    ```nginx
        add_header X-Frame-Options "SAMEORIGIN";
        add_header X-Content-Type-Options "nosniff";
        add_header Referrer-Policy "strict-origin-when-cross-origin";
    ```

    Add this line as the fourth header in the same block:
    ```nginx
        add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; connect-src 'self' https://*.rsgb.online https://*.ukrepeater.net";
    ```

    The result after the change:
    ```nginx
        # Security headers
        add_header X-Frame-Options "SAMEORIGIN";
        add_header X-Content-Type-Options "nosniff";
        add_header Referrer-Policy "strict-origin-when-cross-origin";
        add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; connect-src 'self' https://*.rsgb.online https://*.ukrepeater.net";
    ```

    Placement rationale (per D-07): The header must be in the server block, not inside any location block. nginx add_header directives in a child location block that has its own add_header directives do NOT inherit server-block headers — they replace the inherited set. This means the CSP will NOT appear on static asset responses (the `location ~* \.(js|css|...)$` block has its own `add_header Cache-Control` directive). This is acceptable per D-07: CSP enforcement happens on the HTML document load, not on individual asset responses.

    Do not modify any existing lines. Do not add the header inside any location block.

    The connect-src wildcard directives per D-06:
    - `https://*.rsgb.online` covers current api-beta.rsgb.online and any future subdomain
    - `https://*.ukrepeater.net` covers ukrepeater.net repeater data lookups
    Both wildcards are intentional and documented in D-06 as resilient to subdomain changes on these third-party beta APIs.
